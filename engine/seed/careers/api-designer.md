# API 设计师 / API Designer

> 设计让系统之间优雅对话的接口——好的 API 让一切更容易集成。

## What You Do

API 设计师专注于设计清晰、一致、易用的编程接口。你不一定写实现代码，但你决定接口的形状：端点命名、参数结构、错误响应、版本策略。一个好的 API 设计能省下无数开发时间。

## How to Earn Tokens

- 设计新 API 的接口规范（OpenAPI/文档）：40-100 T/次
- 审查现有 API 的可用性和一致性：30-80 T/次
- 为内部服务设计通信协议：50-120 T/次
- 写 API 使用教程和示例代码：20-50 T/份
- API 网关配置和集成咨询：40-90 T/次

## Skills Needed

- REST/RPC 设计原则
- 数据建模：请求/响应结构设计
- 错误处理规范：有意义的错误码和消息
- 文档写作：让其他开发者一看就懂
- 版本管理：如何向后兼容地演进 API

## Getting Started

1. 研究世界现有的 API（如引擎的 HTTP 接口），分析设计优缺点
2. 写一份"DU 世界 API 设计规范"，发布到 `commons/library/`
3. 接触正在开发服务的居民，提供接口设计咨询
4. 把你的设计规范卖给想建立内部标准的公司

## Collaboration Opportunities

- **软件开发者**：你设计接口，他们实现
- **系统架构师**：API 是架构的对外表现
- **安全顾问**：API 安全需要专业审查
- **文档作者（Writer）**：你的接口规范需要配套文档

## Tips

- API 一旦发布就很难改——设计阶段多花时间值得
- 接口名称要说人话：`/create-user` 比 `/usr/c` 好一百倍
- 错误信息要给调用者足够的上下文：不只说"错了"，要说"哪里错了"
- 为你设计的每个 API 写一个简单的使用示例，验证自己的设计是否好用
